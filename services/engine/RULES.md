# Правила Texas Hold'em в текущем engine

## Базовая модель руки

- Одна раздача соответствует одному `GameState`.
- В раздаче есть:
  - `button`
  - `small blind`
  - `big blind`
  - hole cards у каждого игрока
  - общий board до 5 карт
  - pots
- Улицы идут в таком порядке:
  - `preflop`
  - `flop`
  - `turn`
  - `river`
  - `showdown`

Каждый игрок имеет:
- стек `stack`
- текущую ставку на улице `current_bet`
- статус:
  - `active`
  - `folded`
  - `all-in`
  - `sitting_out`

Игрок может действовать только если он:
- имеет статус `active`
- и его стек больше нуля

## Старт раздачи

Для старта руки engine требует:
- минимум 2 игроков
- корректную позицию `button`
- только `fixed-limit` betting structure
- положительные blinds
- положительные `small_bet` и `big_bet`
- `big_blind == small_blind * 2`
- `small_bet >= big_blind`
- `big_bet >= small_bet`
- `max_raises_per_street > 0`

При старте руки происходит следующее:
- создается и перемешивается колода
- каждому игроку раздается по 2 закрытые карты
- posting blinds происходит сразу
- `current_bet` в состоянии руки становится равен `big_blind`

### Индексы блайндов

Для `heads-up`:
- `small blind` ставит игрок на `button`
- `big blind` ставит второй игрок

Для 3+ игроков:
- `small blind` = следующий после `button`
- `big blind` = следующий после `small blind`

### Кто ходит первым

На `preflop` первым действует первый активный игрок после `big blind`.

На `flop`, `turn`, `river` первым действует первый активный игрок после `button`.

## Порядок улиц

Раздача начинается на `preflop`.

При завершении betting round:
- все текущие ставки игроков собираются в pots
- `current_bet` обнуляется
- `raises_this_street` обнуляется
- назначается новый `active_player`

Дальше улицы продвигаются так:
- из `preflop` в `flop`: на board добавляются 3 карты
- из `flop` в `turn`: на board добавляется 1 карта
- из `turn` в `river`: на board добавляется 1 карта
- из `river` в `showdown`: новые карты не добавляются

## Поддерживаемая структура ставок

Сейчас engine поддерживает только `fixed-limit`.

Размер ставки по улицам:
- на `preflop` и `flop` используется `small_bet`
- на `turn` и `river` используется `big_bet`

Это влияет на допустимый размер `raise`.

## Разрешенные действия

С точки зрения публичного игрового контракта движок сейчас поддерживает:
- `fold`
- `check`
- `call`
- `raise`

В доменной модели существует `all-in`, но как отдельное валидное действие он сейчас не поддержан: `ValidateAction` возвращает `ErrInvalidAction`.

То есть игрок может попасть в статус `all-in` как результат `call` или `raise`, если у него закончился стек, но не через отдельный action `ALL_IN`.

## Правила валидации действий

### Общие правила

Перед проверкой конкретного действия engine требует:
- чтобы ходил именно `active_player`
- чтобы этот игрок мог действовать

Иначе возвращается ошибка.

### Check

`check` разрешен только если игрок уже уравнял текущую ставку.

Иначе:
- если `state.current_bet > player.current_bet`
- `check` запрещен

### Call

`call` разрешен только если текущая ставка выше ставки игрока.

Иначе:
- если `state.current_bet == player.current_bet`
- `call` запрещен

### Raise

`raise` разрешен только если:
- еще не достигнут `max_raises_per_street`
- новый размер ставки точно равен `state.current_bet + BetSizeForStreet()`
- у игрока хватает стека на доведение ставки до этого размера

Важно:
- в engine `raise` задается как `raise-to`
- то есть `action.amount` это не "добавить столько-то", а "довести мою ставку до такого значения"

Пример:
- текущая ставка `100`
- улица `turn`
- `big_bet = 200`
- тогда допустимый `raise-to` равен `300`

Если передан любой другой `amount`, возвращается `ErrInvalidRaiseAmount`.

### Fold

`fold` разрешен как обычное действие текущего активного игрока.

После `fold` статус игрока становится `folded`.

## Применение действий

### Check

`check` не меняет стек и не меняет текущую ставку игрока.

### Call

При `call` engine списывает:
- `state.current_bet - player.current_bet`

Если этой суммы больше или равно оставшемуся стеку игрока:
- списывается весь стек
- игрок получает статус `all-in`

### Raise

При `raise` engine считает:
- `amount = action.amount - player.current_bet`

Потом:
- списывает `amount` со стека
- ставит `player.current_bet = action.amount`
- обновляет `state.current_bet = action.amount`
- увеличивает `raises_this_street`

Если после этого стек игрока равен нулю:
- игрок получает статус `all-in`

## Когда заканчивается betting round

Betting round считается завершенным, когда у каждого игрока, который еще может действовать:
- `player.current_bet >= state.current_bet`

Игроки, которые:
- уже folded
- уже all-in
- не имеют права действовать

не мешают завершению betting round.

## Когда заканчивается раздача

Раздача заканчивается в двух случаях.

### 1. Остался один игрок, который еще может действовать

Если `len(PlayersWhoCanAct()) <= 1`, engine завершает руку сразу.

Практически это покрывает типичный сценарий, когда все остальные сфолдили.

### 2. Дошли до showdown

Если после `river` betting round завершен, улица переводится в `showdown`, и дальше запускается завершение руки.

## Pots и side pots

Во время торговли деньги живут в `player.current_bet`.

Когда улица заканчивается или рука завершается:
- engine собирает текущие ставки всех игроков
- строит pots из contributions

Каждый pot содержит:
- `amount`
- список `eligible` игроков, которые могут его выиграть

### Side pots

Side pots поддерживаются.

Расчет строится по минимальному оставшемуся вкладу среди игроков с ненулевыми contributions:
- создается pot на `min_contrib * число_игроков`
- затем этот минимум вычитается из вкладов
- процесс повторяется, пока contributions не закончатся

Если pots уже существовали с предыдущих улиц:
- первый новый pot мержится в последний уже существующий
- остальные pots добавляются отдельно

Это поведение важно, потому что engine копит банк по улицам, а не пересчитывает весь hand pot с нуля каждый раз.

## Showdown и определение победителей

Если в руке остался только один активный участник:
- он получает весь общий pot
- showdown evaluator не нужен

Если активных участников больше одного:
- для каждого игрока evaluator ищет лучшую 5-карточную комбинацию из 7 карт
- дальше для каждого pot отдельно выбирается лучший результат среди `eligible` игроков

Победители определяются отдельно для каждого pot.

### Split pot

Если в одном pot несколько победителей:
- сумма pot делится поровну целой частью
- остаток распределяется по первым победителям в порядке обхода winners для этого pot

После расчета winnings деньги просто добавляются обратно в `player.stack`.

## Rake

Rake считается только в конце руки.

Конфиг рейка состоит из:
- `percent`
- `cap`
- `no_flop_no_drop`

### No flop, no drop

Если включен `no_flop_no_drop` и рука закончилась на `preflop`:
- rake = 0

### Обычный расчет

Иначе:
- берется `total(pots)`
- считается `int(total * percent)`
- затем применяется ограничение `cap`

### Как rake снимается с pots

Rake вычитается не из абстрактной суммы, а из массива pots:
- начиная с последнего pot
- пока весь rake не будет списан

Из-за этого:
- `gross_pot` это pot до rake
- `net_pot` это pot после rake

## Ограничения текущей реализации

Ниже перечислены не "особенности покера", а именно ограничения текущего engine.

- Поддерживается только `fixed-limit`.
- Отдельное действие `ALL_IN` не поддержано, хотя внутренний статус `all-in` существует.
- Документ описывает текущую реализацию engine, а не полный live poker ruleset.
- Правила про burn cards, dead button, missed blinds, rebuys, tournament-specific house rules и другие реальные room rules здесь не зафиксированы, потому что они сейчас не выражены в коде.

## Связь с event model

Эта игровая логика напрямую отражается в событиях engine:
- `hand.started` фиксирует игроков, button, betting structure и blinds
- `hand.acted` фиксирует street, action, current bet и ставку игрока после применения действия
- `hand.ended` фиксирует showdown flag, gross/net pot, rake, board и winners

Порядок событий внутри руки задается `sequence_number`.

Поэтому `RULES.md` описывает не только поведение игры, но и семантику данных, которые потом уходят в Kafka, ClickHouse и аналитический слой.
